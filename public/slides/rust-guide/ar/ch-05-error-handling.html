Error Handling

تعامل مع الأخطاء بأناقة باستخدام Result و Option وعامل ? وأنواع الأخطاء المخصصة.

1

panic! للأخطاء غير الRecoverable

يتطلب منك Rust الاعتراف بإمكانية الفشل والتعامل معه. هذا النهج الصريح يمنع الأخطاء التي يتم تجاهلها بصمت. يقسّم Rust الأخطاء إلى فئتين: Recoverable وUnrecoverable.

عندما يحدث خطأ كارثي ولا توجد طريقة للاسترداد، استخدم panic!. ينهي البرنامج فوراً:

Rust
fn main() {
    panic!("crash and burn");  // Program terminates here
}

غالباً ما تنتج حالات panic من الوصول إلى فهارس مصفوفة خارج الحدود أو فك تغليف قيم None:

Rust
fn main() {
    let v = vec![1, 2, 3];
    // This will panic - index out of bounds
    // println!("{}", v[99]);
    // This will panic - None value
    let option: Option<i32> = None;
    // let value = option.unwrap();  // Panic!
}
💡

متى تستخدم panic!

استخدم panic! فقط في الحالات غير الRecoverable فعلاً. لمعظم حالات الأخطاء، استخدم Result<T, E> بدلاً من ذلك.

2

Result<T, E> للأخطاء الRecoverable

تعداد Result يحتوي على Variableين وهو الطريقة القياسية في Rust للتعامل مع الأخطاء الRecoverable:

Rust
enum Result<T, E> {
    Ok(T),     // Operation succeeded with value T
    Err(E),    // Operation failed with error E
}

معالجة Result الأساسية

Rust
use std::fs::File;
fn main() {
    let f = File::open("hello.txt");
    match f {
        Ok(file) => println!("File opened successfully"),
        Err(error) => println!("Error opening file: {:?}", error),
    }
}

unwrap و expect

unwrap() تُرجع قيمة Ok أو تسبب panic. أما expect() فتفعل الشيء نفسه لكن مع رسالة خطأ مخصصة:

Rust
use std::fs::File;
fn main() {
    let f = File::open("hello.txt")
        .expect("Failed to open hello.txt");  // Panic with message if error
    // Or use unwrap (less informative)
    // let f = File::open("hello.txt").unwrap();
}
⚠️

تحذير

تجنب unwrap() في كود الإنتاج. ستُعطّل برنامجك بدون رسالة خطأ مفيدة. يُفضَّل استخدام expect() مع رسالة وصفية، أو معالجة الخطأ صراحةً.

3

تمرير الأخطاء باستخدام ?

يُبسّط عامل ? تمرير الأخطاء. إذا أرجعت الـ Function خطأً، فإن ? يُعيد ذلك الخطأ من الـ Function الحالية:

Rust
use std::fs::File;
use std::io;
use std::io::Read;
fn read_username_from_file() -> Result<String, io::Error> {
    let mut f = File::open("username.txt")?;  // Propagate error
    let mut s = String::new();
    f.read_to_string(&mut s)?;  // Propagate error
    Ok(s)
fn main() {
    match read_username_from_file() {
        Ok(username) => println!("Username: {}", username),
        Err(error) => println!("Error: {:?}", error),
    }
}

تسلسل النتائج

استخدم دوال مثل and_then و map للتعامل مع Result بأسلوب وظيفي:

Rust
fn parse_and_square(s: &str) -> Result<i32, std::num::ParseIntError> {
    s.parse::<i32>()
        .map(|n| n * n)  // Transform Ok value
fn main() {
    match parse_and_square("5") {
        Ok(result) => println!("Result: {}", result),  // Prints: 25
        Err(e) => println!("Error: {}", e),
    }
}
✨

ملاحظة مهمة

عامل ? هو اختصار يُغني عن كتل match المطوّلة. يقوم تلقائياً بتحويل نوع الخطأ إذا كان trait الـ From مُنفَّذاً، مما يجعل تمرير الأخطاء نظيفاً وقابلاً للتركيب.

4

أنواع الأخطاء المخصصة

للتطبيقات المعقدة، عرّف أنواع أخطاء مخصصة:

Rust
use std::fmt;
struct CustomError {
    message: String,
impl fmt::Display for CustomError {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "Custom Error: {}", self.message)
    }
impl fmt::Debug for CustomError {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        write!(f, "{{ file: {}, line: {} }}", file!(), line!())
    }
fn main() {
    let error = CustomError {
        message: String::from("Something went wrong"),
    };
    println!("{}", error);    // Uses Display
    println!("{:?}", error);  // Uses Debug
}
5

حزمتا thiserror و anyhow

لكود الإنتاج، استخدم حزم Error Handling الشائعة. أولاً، أضفها إلى ملف Cargo.toml:

TOML
[dependencies]
thiserror = "1.0"
anyhow = "1.0"

استخدام thiserror

تُسهّل حزمة thiserror تعريف أنواع أخطاء مخصصة:

Rust
use thiserror::Error;
#[derive(Error, Debug)]
pub enum DataError {
    #[error("IO error: {0}")]
    Io(#[from] std::io::Error),
    #[error("Parse error: {0}")]
    Parse(#[from] std::num::ParseIntError),
    #[error("Invalid data: {0}")]
    Invalid(String),
fn process_data(s: &str) -> Result<i32, DataError> {
    s.parse()
        .map_err(|_| DataError::Invalid(String::from("Not a number")))
fn main() {
    match process_data("not_a_number") {
        Ok(num) => println!("Parsed: {}", num),
        Err(e) => eprintln!("Error: {}", e),
    }
}

استخدام anyhow

توفر حزمة anyhow معالجة أخطاء مرنة للتطوير السريع:

Rust
use anyhow::{Result, Context};
use std::fs;
fn read_config(path: &str) -> Result<String> {
    let contents = fs::read_to_string(path)
        .context("Failed to read config file")?;
    Ok(contents)
fn main() {
    match read_config("config.txt") {
        Ok(config) => println!("Config: {}", config),
        Err(e) => eprintln!("Error: {}", e),
    }
}
💡

نصيحة

استخدم thiserror للمكتبات (أخطاء مُهيكلة ومُنمَّطة) و anyhow للتطبيقات (معالجة أخطاء مرنة وسريعة). تُكمّلان بعضهما جيداً.

6

نمط عملي لError Handling

إليك مثالاً عملياً يجمع عدة مفاهيم:

Rust
use std::fs::File;
use std::io::{self, Read};
fn read_and_parse_file(path: &str) -> Result<i32, Box<dyn std::error::Error>> {
    // Open file, propagating IO errors
    let mut file = File::open(path)?;
    // Read contents, propagating IO errors
    let mut contents = String::new();
    file.read_to_string(&mut contents)?;
    // Parse to integer, propagating parse errors
    let number = contents.trim().parse::<i32>()?;
    Ok(number)
fn main() {
    match read_and_parse_file("number.txt") {
        Ok(num) => println!("Number: {}", num),
        Err(e) => {
            eprintln!("An error occurred: {}", e);
            std::process::exit(1);
        }
    }
}

أفضل ممارسات Error Handling

عند تصميم Error Handling لتطبيقك:

  • استخدم Result افتراضياً: Functions التي قد تفشل يجب أن تُرجع Result.
  • مرّر الأخطاء للأعلى: استخدم عامل ? لتترك المُستدعي يقرر كيفية التعامل مع الأخطاء.
  • وفّر السياق: أضف معلومات حول ما فشل ولماذا.
  • استخدم أخطاء مخصصة للمكتبات: يجب أن تُعرّف المكتبات أنواع أخطائها الخاصة.
  • استخدم anyhow للتطبيقات: يمكن للتطبيقات استخدام معالجة أخطاء مرنة مع anyhow.
  • تجنب unwrap في الإنتاج: استخدم unwrap() فقط عندما تكون متأكداً تماماً من وجود القيمة.
  • سجّل الأخطاء: استخدم حزمة log أو ما شابهها لتسجيل الأخطاء لأغراض التصحيح.
✨

ملاحظة مهمة

فلسفة Error Handling في Rust هي "اجعل الحالات غير الصالحة غير قابلة للتمثيل." بترميز الفشل في Type System باستخدام Result و Option، يكتشف المُترجم Error Handling المفقودة في Compile-time, وليس في Runtime.
