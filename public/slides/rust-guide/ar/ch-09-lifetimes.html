Lifetimes

Lifetimes بوضوح, كيف يتتبع Rust صلاحية الـ References في Compile-time.

1

لماذا توجد Lifetimes

تمنع Lifetimes الـ References المعلّقة, المؤشرات إلى بيانات تم تحريرها. يستخدم Rust Compiler Lifetime Annotations للتحقق من أن كل Reference صالح طوال فترة استخدامه.

Rust
// Problem: dangling references
// fn dangle() -> &String {
//     let s = String::from("hello");
//     &s  // ERROR: s goes out of scope, reference becomes invalid
// }
// Solution: return owned value
fn no_dangle() -> String {
    let s = String::from("hello");
    s  // Ownership transferred to caller
// Lifetime annotation tells compiler how long references are valid
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
let s1 = String::from("long string");
let s2 = "short";
println!("{}", longest(&s1, s2));

let first = get_first(&strs);
⚠️

Lifetimes لا تُغيّر مدة بقاء البيانات

Lifetime Annotations لا تُمدد أو تُقصّر العمر الفعلي لأي بيانات. إنها طريقة لإخبار الـ Compiler عن العلاقات بين فترات حياة الـ References المختلفة، حتى يتمكن من التحقق من الصحة في Compile-time.

2

صيغة Lifetime Annotations

تُعلَن معاملات Lifetimes بين أقواس زاوية بعد اسم الـ Function، باستخدام علامة اقتباس أحادية متبوعة بحرف صغير: 'a، 'b، إلخ. تصف كيف ترتبط فترات حياة المدخلات والمخرجات.

Rust
// Single lifetime parameter
fn take_ref<'a>(s: &'a String) -> &'a str {
    &s[..]
// Multiple lifetime parameters
fn compare<'a, 'b>(x: &'a str, y: &'b str) -> bool {
    x.len() == y.len()
// Lifetime bounds: 'b must live at least as long as 'a
fn subset<'a, 'b: 'a>(x: &'a str, y: &'b str) -> &'a str {
    if x.len() < y.len() {
        x
    } else {
        y  // y lives longer, can be returned as 'a
    }
}
Rust

قيود Lifetimes مع Where

الصيغة 'b: 'a تعني "'b تعيش أطول من 'a". هذا يتيح لك إرجاع Reference بفترة حياة 'a من قيمة لها فترة حياة أطول 'b. استخدم Statements where للExpression عن هذه القيود بوضوح.

3

Lifetime Elision Rules

يمتلك Rust ثلاث قواعد لحذف Lifetimes (lifetime elision) تتيح للCompiler استنتاج Lifetimes في الـ Patterns الشائعة، فلا تحتاج لكتابتها صراحة في كل مرة.

Rust
// Rule 1: Each input lifetime gets its own lifetime parameter
// These are equivalent:
fn input_ref<'a>(x: &'a str) -> String {
    String::from(x)
fn input_ref(x: &str) -> String {
    String::from(x)
// Rule 2: If there's one input lifetime, it's assigned to output
// These are equivalent:
fn only_input<'a>(x: &'a str) -> &'a str {
    x
fn only_input(x: &str) -> &str {
    x
// Rule 3: If there's &self, self's lifetime applies to output
struct Container;
impl Container {
    // Implicit: fn first<'a>(&'a self) -> &'a str
    fn first(&self) -> &str {
        "hello"
    }
}
Rust
// When elision rules don't apply, explicit annotation required
fn two_inputs<'a, 'b>(x: &'a str, y: &'b str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        x  // Must choose one lifetime
    }
// Compiler can't elide this - which lifetime for output?
// fn ambiguous(x: &str, y: &str) -> &str {
//     if true { x } else { y }
// }
// Method with explicit lifetimes
struct Parser<'a> {
    input: &'a str,
impl<'a> Parser<'a> {
    fn parse(&self) -> &'a str {
        self.input
    }
    fn parse_ref<'b>(&'b self) -> &'b str {
        self.input
    }
}
✨

الحذف يُبقي الكود نظيفاً

عملياً، تغطي Elision Rules الثلاث الغالبية العظمى من الحالات. تحتاج فقط إلى تعليقات فترات حياة صريحة عندما لا يستطيع الـ Compiler تحديد العلاقة بين فترات حياة المدخلات والمخرجات, عادةً عندما تكون هناك عدة معاملات Reference ونوع إرجاع Referenceي.

متى تستخدم فترات حياة متعددة

استخدم فترة حياة واحدة عندما يجب أن تأتي جميع الـ References في الهيكل من بيانات بنفس الـ Scope. استخدم فترات حياة متعددة (مثل 'a و 'b) عندما قد تأتي الـ References من بيانات بنطاقات مختلفة، مما يمنح الـ Compiler مرونة أكبر.

Lifetime الساكنة وقيود Lifetimes

Lifetime 'static خاصة, تعني أن البيانات تعيش طوال مدة البرنامج بأكملها. النصوص الحرفية والأنواع المملوكة بدون مراجع مستعارة تستوفي 'static.

// 'static bound: type must only contain 'static references
fn print_static<T: 'static>(val: T) {
    // T doesn't contain borrowed references
print_static(String::from("hello"));
print_static(5);
// print_static(&String::from("hello")); // Error: not 'static
// Using 'static with trait objects

    fn do_something(&self);

    fn do_something(&self) {
        println!("Doing something");
    }
fn get_trait_object() -> Box<dyn MyTrait + 'static> {
    Box::new(MyType)
}

لا تُفرط في استخدام 'static

Box::leak يُسرّب الـ Memory عمداً ويجب استخدامه فقط للبيانات التي تحتاج حقاً للبقاء للأبد (مثل التكوين العام). في معظم الحالات، فضّل Lifetime Annotations المناسبة أو الأنواع المملوكة بدلاً من قيود 'static.

Trait Bounds عالية الرتبة (HRTBs)

صيغة for<'a> تعني "لأي فترة حياة 'a". تُستخدم عندما يجب أن يعمل إغلاق أو Function مع مراجع من أي فترة حياة، وليس فترة محددة فقط. ستصادف هذا غالباً مع Closures المُمررة كمعاملات.

الـ Patterns الشائعة والأخطاء المتكررة لLifetimes

فهم الـ Patterns والأخطاء الشائعة يساعدك على كتابة Lifetime Annotations الصحيحة بسرعة وتجنب أخطاء الـ Compiler المحبطة.

// Pitfall 1: Returning reference to local data
// fn pitfall1() -> &str {
//     let s = String::from("hello");

أفضل ممارسات Lifetimes

استخدم 'a لLifetime الأساسية، و 'b و 'c للإضافية. أبقِ Lifetime Annotations في حدها الأدنى وواضحة. فضّل الأنواع المملوكة عند الإمكان لتجنب تعقيد Lifetimes. استخدم 'static فقط للبيانات التي تعيش فعلاً لفترة طويلة.
