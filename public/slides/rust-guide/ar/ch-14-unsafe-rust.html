Rust غير الآمن (unsafe)

متى وكيف تستخدم unsafe, Raw Pointers و FFI وتجاوز الـ Borrow Checker بأمان.

1

متى ولماذا تستخدم unsafe

يتجاوز Rust غير الآمن (unsafe) ضمانات وقت التجميع عندما يكون ذلك ضرورياً للأداء أو العمليات على مستوى النظام. استخدمه باعتدال ومع توثيق دقيق.

Rust
// When performance requires unsafe optimizations
fn summing_without_bounds_checking(slice: &[i32]) -> i32 {
    let mut sum = 0;
    unsafe {
        // Bypassing bounds checking when we know it's safe
        let ptr = slice.as_ptr();
        for i in 0..slice.len() {
            sum += *ptr.add(i);
        }
    }
    sum
fn main() {
    let numbers = vec![1, 2, 3, 4, 5];
    let result = summing_without_bounds_checking(&numbers);
    println!("Sum: {}", result);
}
⚠️

عقد unsafe

داخل كتلة unsafe، أنت المسؤول عن الحفاظ على ثوابت أمان Rust. الـ Compiler يثق بك, وإذا أخطأت، ستحصل على سلوك غير محدد.

2

Raw Pointers

تتيح Raw Pointers (*const T و*mut T) الوصول المباشر للذاكرة بدون فحوصات أمان. إنشاؤها آمن، لكن إلغاء Referenceيتها يتطلب كتلة unsafe.

Rust
fn main() {
    let x = 5;
    let raw_ptr = &x as *const i32;
    // Can only dereference in unsafe blocks
    unsafe {
        println!("Value at pointer: {}", *raw_ptr);
    }
    // Mutable raw pointers
    let mut y = 10;
    let mut_ptr = &mut y as *mut i32;
    unsafe {
        *mut_ptr = 20;
        println!("Modified value: {}", *mut_ptr);
    }
}
Rust
// Pointer arithmetic - advanced usage
fn main() {
    let arr = [1, 2, 3, 4, 5];
    let ptr = arr.as_ptr();
    unsafe {
        println!("First element: {}", *ptr);
        println!("Third element: {}", *ptr.add(2));
        // Copy from pointer
        let mut dest = [0; 5];
        std::ptr::copy_nonoverlapping(ptr, dest.as_mut_ptr(), 5);
        println!("Copied array: {:?}", dest);
    }
}
💡

الإنشاء مقابل إلغاء الـ Reference

يمكنك إنشاء مؤشرات خام في كود آمن, وهذا مسموح تماماً. فقط إلغاء Reference المؤشر الخام هو ما يتطلب unsafe، لأن الـ Compiler لا يمكنه ضمان صلاحية المؤشر.

3

Functions والكتل غير الآمنة

إنشاء واستدعاء Functions غير الآمنة يتطلب كتل unsafe صريحة. الـ Pattern الجيد هو تغليف العمليات الداخلية غير الآمنة خلف واجهة برمجية عامة آمنة.

Rust
// Unsafe function that splits a string at an arbitrary position
unsafe fn split_at_unsafe(s: &str, mid: usize) -> (&str, &str) {
    let bytes = s.as_bytes();
    let ptr = s.as_ptr();
    // Assume mid is a valid UTF-8 boundary
    (
        std::str::from_utf8_unchecked(
            std::slice::from_raw_parts(ptr, mid)),
        std::str::from_utf8_unchecked(
            std::slice::from_raw_parts(ptr.add(mid), bytes.len() - mid)),
    )
fn safe_split_wrapper(s: &str, mid: usize) -> Option<(&str, &str)> {
    if mid <= s.len() && s.is_char_boundary(mid) {
        unsafe { Some(split_at_unsafe(s, mid)) }
    } else {
        None
    }
fn main() {
    let text = "Hello, world!";
    if let Some((left, right)) = safe_split_wrapper(text, 5) {
        println!("Left: {}, Right: {}", left, right);
    }
}
Rust
// Unsafe traits
unsafe trait Dangerous {
    unsafe fn do_dangerous_thing(&self);
struct UnsafeStruct;
unsafe impl Dangerous for UnsafeStruct {
    unsafe fn do_dangerous_thing(&self) {
        println!("Doing something dangerous!");
    }
fn main() {
    let obj = UnsafeStruct;
    unsafe {
        obj.do_dangerous_thing();
    }
}

نمط الأغلفة الآمنة

أفضل ممارسة هي التحقق من جميع الشروط المسبقة في Function غلاف آمنة، ثم استدعاء كود unsafe الأساسي فقط عندما تكون الثوابت مضمونة. هذا يبقي مساحة unsafe في حدها الأدنى.

4

Variables الثابتة القابلة للتعديل

تتطلب Variables الثابتة القابلة للتعديل (mutable statics) وصولاً غير آمن بسبب احتمال حدوث سباقات بيانات في السياقات Multi-threaded.

Rust
static mut COUNTER: i32 = 0;
fn increment_counter() {
    unsafe {
        COUNTER += 1;
    }
fn get_counter() -> i32 {
    unsafe {
        COUNTER
    }
fn main() {
    increment_counter();
    increment_counter();
    println!("Counter: {}", get_counter());
}
⚠️

فضّل Atomics أو Mutex

Variables الثابتة القابلة للتعديل غير آمنة بطبيعتها في الكود متعدد Threads. في الممارسة العملية، فضّل AtomicI32 أو Mutex<T> أو once_cell::sync::Lazy لإدارة الحالة العامة بThread Safety.

5

FFI (FFI): استدعاء C من Rust

تتيح FFI (FFI) استدعاء كود C من Rust. جميع استدعاءات FFI هي unsafe لأن Rust Compiler لا يمكنه التحقق من صحة الكود الخارجي.

Rust
// Using libc to call C functions
extern "C" {
    fn abs(x: i32) -> i32;
    fn sqrt(x: f64) -> f64;
fn main() {
    unsafe {
        println!("abs(-42) = {}", abs(-42));
        println!("sqrt(16.0) = {}", sqrt(16.0));
    }
}
Rust
// Wrapping C functions with safe interfaces
extern "C" {
    fn strlen(s: *const u8) -> usize;
fn safe_strlen(s: &str) -> usize {
    unsafe {
        strlen(s.as_ptr())
    }
fn main() {
    let text = "Hello";
    println!("Length: {}", safe_strlen(text));
}
💡

واجهة ABI من النوع extern "C"

حرف "C" في extern "C" يحدد اتفاقية الاستدعاء. هذا يخبر Rust باستخدام ABI لغة C، وهو المعيار للتوافق بين اللغات على معظم المنصات.

6

تجريدات آمنة فوق كود unsafe

أفضل ممارسة هي إخفاء العمليات غير الآمنة خلف واجهات برمجية آمنة. تحقق من المدخلات، وأدِر Lifetimes، ونفّذ Drop للتنظيف.

Rust
// Safe abstraction over unsafe pointer operations
struct SafeBuffer {
    ptr: *mut u8,
    len: usize,
impl SafeBuffer {
    fn new(size: usize) -> Self {
        unsafe {
            let ptr = libc::malloc(size) as *mut u8;
            SafeBuffer { ptr, len: size }
        }
    }
    fn get(&self, index: usize) -> Option<u8> {
        if index < self.len {
            unsafe { Some(*self.ptr.add(index)) }
        } else {
            None
        }
    }
    fn set(&mut self, index: usize, value: u8) -> bool {
        if index < self.len {
            unsafe { *self.ptr.add(index) = value; }
            true
        } else {
            false
        }
    }
impl Drop for SafeBuffer {
    fn drop(&mut self) {
        unsafe {
            libc::free(self.ptr as *mut libc::c_void);
        }
    }
fn main() {
    let mut buf = SafeBuffer::new(10);
    buf.set(0, 42);
    if let Some(value) = buf.get(0) {
        println!("Value: {}", value);
    }
}
✨

المبدأ الأساسي

التجريد الآمن المصمم جيداً يجعل من المستحيل على المستدعين إثارة سلوك غير محدد من خلال الواجهة البرمجية العامة. جميع فحوصات الحدود وإثباتات الصلاحية تحدث داخل الغلاف.

7

Patterns unsafe الشائعة

تتضمن عدة Patterns في برمجة الأنظمة كوداً غير آمن بشكل شائع، مثل تحويل الأنواع واستخدام هياكل #[repr(C)] للتوافق مع C.

Rust
// Pattern 1: Transmute (with extreme caution)
fn bytes_to_u32_unsafe(bytes: &[u8; 4]) -> u32 {
    unsafe {
        std::mem::transmute::<[u8; 4], u32>(*bytes)
    }
// Pattern 2: Accessing struct fields at offsets
#[repr(C)]
struct CStruct {
    a: i32,
    b: f64,
fn main() {
    let s = CStruct { a: 42, b: 3.14 };
    // Safe approach: use normal field access
    println!("a: {}, b: {}", s.a, s.b);
    // Unsafe approach: raw pointer access
    unsafe {
        let ptr = &s as *const CStruct as *const u8;
        println!("Field a offset: {}",
            std::mem::offset_of!(CStruct, a));
    }
}
⚠️

transmute خطير

std::mem::transmute يعيد تفسير بتات القيمة كنوع مختلف. يمكن أن يسبب سلوكاً غير محدد بسهولة إذا كانت للأنواع المصدر والهدف ثوابت مختلفة. فضّل from_ne_bytes / to_ne_bytes للتحويلات الرقمية.
