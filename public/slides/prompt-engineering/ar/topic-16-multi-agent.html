أنظمة الوكلاء المتعددين

عندما لا يكفي وكيل واحد — نظّم عدة وكلاء متخصصين لمعالجة المهام المعقدة. افهم أنماط التعاون التي تحل مشاكل لا يستطيع وكيل واحد حلها.

1

لماذا الوكلاء المتعددون؟ قوة التخصص

الوكيل الواحد يكون متعدد المهارات لكن لا يتقن أياً منها بعمق. نظام الوكلاء المتعددين يخصص أدواراً محددة: وكيل يبحث، وآخر يحلل، وثالث يكتب. تماماً كفريق تطوير يعمل على منتج.

👷

تشبيه: فريق تطوير

مطور واحد يمكنه بناء تطبيق، لكن الفريق أفضل: مهندس الواجهة يتولى واجهة المستخدم، ومهندس الخلفية يتولى الـ API، ومهندس DevOps يتولى النشر. كل واحد يتخصص، ثم يتكاملون. أنظمة الوكلاء المتعددين تعمل بنفس الطريقة: كل وكيل خبير، وهم ينسّقون معاً.

🎯

تقسيم العمل

المهام المعقدة تُقسّم إلى مهام فرعية متخصصة. كل وكيل يتفوق في شيء واحد.

⚡

العمل المتوازي

الوكلاء يعملون بالتوازي (أو بالتسلسل) لتسريع إنجاز المهمة الكلية.

🔍

تحسين الجودة

مخرجات وكيل تصبح مدخلات لوكيل آخر. التحسين التكراري يرفع الجودة.

🛡️

كشف الأخطاء

وكيل ناقد يمكنه التحقق من عمل الوكلاء الآخرين، ويلتقط الأخطاء قبل التسليم.

2

أنماط الوكلاء المتعددين — أربع بنيات رئيسية

هناك أربعة أنماط مثبتة لتنسيق وكلاء متعددين. كل نمط يناسب أنواعاً مختلفة من المهام.

النمط 1: المشرف (المدير)

وكيل مشرف واحد يفوّض المهام للمتخصصين. المشرف يحلل الهدف، ويوزّع المهام الفرعية، ويجمع النتائج، ويدمجها. هذا هو نمط التنسيق الكلاسيكي.

Text — Supervisor Pattern Flow 
User: "Write a product launch plan"
       ↓
Supervisor Agent:
  - Breaks into: Market Analysis, Budget Planning, Timeline, Risk Mitigation
  - Assigns to specialists
       ↓
Market Analyst: "Competitors, target audience, positioning"
Budget Planner: "Cost breakdown, revenue projections"
Timeline Manager: "Phases, milestones, dependencies"
Risk Analyst: "Risks, mitigation strategies"
       ↓
Supervisor: Gathers results, synthesizes into final plan
       ↓
User: Complete launch plan

النمط 2: نظير إلى نظير

الوكلاء يعملون معاً على نفس المستوى. كل وكيل يمكنه بدء رسائل للآخرين. مفيد للمشاكل الاستكشافية حيث الحل ينبثق من النقاش.

النمط 3: خط الأنابيب

مخرجات وكيل تصبح مدخلات للوكيل التالي. سير عمل خطي. باحث ← محلل ← كاتب ← محرر. تدفق بيانات واضح.

النمط 4: المناظرة / الناقد

عدة وكلاء يدافعون عن وجهات نظر مختلفة، وحكم/ناقد يقرر. ممتاز لتحسين الجودة من خلال كشف نقاط الضعف في الاستدلال.

👔

المشرف (هرمي)

منسّق واحد، عدة متخصصين. الأفضل لـ: المهام المحددة ذات المهام الفرعية الواضحة.

🤝

نظير إلى نظير (مسطّح)

جميع الوكلاء متساوون، ينسّقون ذاتياً. الأفضل لـ: العمل الاستكشافي والتعاوني.

⛓️

خط الأنابيب (تسلسلي)

سلسلة مخرجات ← مدخلات. الأفضل لـ: سير العمل القائم على التحويل (بحث ← تحليل ← كتابة).

⚖️

المناظرة/الناقد (تنافسي)

وجهات نظر متعارضة، حكم يقرر. الأفضل لـ: تحسين الجودة، القرارات الصعبة.

3

بروتوكولات التواصل — كيف تتحدث الوكلاء

الوكلاء يحتاجون طريقة للتواصل: تمرير البيانات، طلب المعلومات، تنسيق الإجراءات. هناك عدة مقاربات، لكل منها مقايضات.

💬

تمرير الرسائل

الوكلاء يرسلون رسائل منظمة لبعضهم. غير متزامن وقليل الارتباط. مناسب لخطوط الأنابيب.

📦

الحالة المشتركة

جميع الوكلاء يقرأون ويكتبون في قاموس حالة مشترك. مترابط بشدة لكن سريع. خطر التعارضات.

⚫

نمط السبورة

"سبورة" مركزية ينشر عليها الوكلاء اكتشافاتهم. الآخرون يقرأون ويبنون على الأفكار. رائع للعمل الاستكشافي.

🗂️

قاعدة بيانات/نظام ملفات

تخزين دائم للنتائج الوسيطة. يصمد أمام إعادة تشغيل الوكلاء. الأفضل للإنتاج.

4

نمط المشرف — تطبيق كامل

لننفّذ نمط المشرف في بايثون. هذا هو النمط الأكثر عملية لأنظمة الإنتاج.

Python — Supervisor Pattern 
class Agent:
    def __init__(self, name: str, role: str):
        self.name = name
        self.role = role
        self.client = anthropic.Anthropic()

    def work(self, task: str) -> str:
        """Execute a task and return result."""
        response = self.client.messages.create(
            model="claude-opus-4-6",
            max_tokens=2048,
            system=f"You are a {self.role}. Your name is {self.name}. Do excellent work.",
            messages=[{"role": "user", "content": task}]
        )
        return response.content[0].text

class SupervisorAgent:
    def __init__(self):
        self.supervisor = Agent("Supervisor", "Project Manager")
        self.specialists = [
            Agent("Alice", "Market Research Analyst"),
            Agent("Bob", "Financial Analyst"),
            Agent("Carol", "Project Manager")
        ]

    def run(self, goal: str) -> str:
        """Supervisor breaks goal into subtasks, delegates to specialists."""

        # 1. Supervisor analyzes goal and creates subtasks
        subtasks_prompt = f"""Goal: {goal}

Break this into 3 specific subtasks for different specialists:
1. For a Market Research Analyst
2. For a Financial Analyst
3. For a Project Manager

Format:
SUBTASK 1: [task for analyst 1]
SUBTASK 2: [task for analyst 2]
SUBTASK 3: [task for analyst 3]"""

        subtasks_text = self.supervisor.work(subtasks_prompt)

        # 2. Parse subtasks (simple regex approach)
        subtasks = []
        for line in subtasks_text.split("\n"):
            if "SUBTASK" in line:
                subtasks.append(line)

        # 3. Distribute to specialists
        results = []
        for i, specialist in enumerate(self.specialists):
            if i < len(subtasks):
                result = specialist.work(subtasks[i])
                results.append({
                    "specialist": specialist.name,
                    "result": result
                })

        # 4. Supervisor synthesizes results
        synthesis_prompt = f"""You have collected the following results from specialists:

{chr(10).join([f"{r['specialist']}: {r['result'][:300]}" for r in results])}

Synthesize these into a cohesive final deliverable for the goal: {goal}

Provide a comprehensive, well-structured final answer."""

        final_answer = self.supervisor.work(synthesis_prompt)
        return final_answer

# Usage
supervisor = SupervisorAgent()
result = supervisor.run("Create a launch strategy for a new SaaS product")
print(result)

نمط المناظرة — الجودة من خلال التعاون التنافسي

وكيلان يتناظران من موقفين متعاكسين. حكم يقيّم كليهما ويقرر. هذا يحسّن جودة الإجابة بإجبار الاستدلال على كشف نقاط الضعف.

Python — Debate Pattern 
class DebateAgents:
    def __init__(self):
        self.pro_agent = Agent("Pro", "Advocates FOR the proposal")
        self.con_agent = Agent("Con", "Advocates AGAINST the proposal")
        self.judge = Agent("Judge", "Evaluates both sides fairly")

    def debate(self, question: str) -> str:
        # Round 1: Pro argues FOR
        pro_argument = self.pro_agent.work(
            f"Argue STRONGLY FOR: {question}\nBe compelling and data-driven."
        )

        # Round 2: Con argues AGAINST
        con_argument = self.con_agent.work(
            f"Argue STRONGLY AGAINST: {question}\nAddress Pro's points if possible."
        )

        # Round 3: Judge decides
        judgment = self.judge.work(
            f"""Question: {question}

Pro argument: {pro_argument}

Con argument: {con_argument}

Evaluate both arguments. Which is stronger? Why? What's the best decision?"""
        )

        return f"""
PRO ARGUMENT:
{pro_argument}

CON ARGUMENT:
{con_argument}

JUDGE'S DECISION:
{judgment}"""

# Usage
debate = DebateAgents()
decision = debate.debate("Should we adopt microservices architecture?")
print(decision)

متى تفوز المناظرة

القرارات الاستراتيجية، خيارات البنية التقنية، قرارات التوظيف. عندما يكون للإجابة إيجابيات وسلبيات مشروعة، المناظرة التي تكشف كلا الجانبين تحسّن الجودة.

6

تحديات أنظمة الوكلاء المتعددين

أنظمة الوكلاء المتعددين قوية لكنها تقدم مشاكل جديدة. فهم التحديات أمر حاسم للنجاح.

⚠️

التحدي 1: عبء التنسيق

وكلاء أكثر = تواصل أكثر. تمرير الرسائل، مشاركة السياق، إدارة الحالة. للمهام البسيطة، العبء يفوق الفوائد. وكيل واحد قد يكون أسرع.

⚠️

التحدي 2: الحلقات اللانهائية

الوكيل أ يسأل الوكيل ب، الذي يسأل الوكيل ج، الذي يسأل الوكيل أ. الحلقات غير المضبوطة تهدر التوكنات والوقت. أضف شروط إنهاء صارمة وكشف الحلقات.

⚠️

التحدي 3: المخرجات المتناقضة

وكلاء مختلفون يختلفون أو يتناقضون. تحتاج حكماً أو وسيطاً. ابنِ آلية حل النزاعات في تصميم نظامك.

⚠️

التحدي 4: انفجار التكلفة

وكلاء متعددون = استدعاءات LLM متعددة. 5 وكلاء × 10 تكرارات = 50 استدعاء API. التكاليف تتراكم بسرعة. راقب وخطط الميزانية بعناية.

7

متى تكون الوكلاء المتعددون مبالغة؟

ليست كل مشكلة تحتاج وكلاء متعددين. أحياناً وكيل واحد مصمم جيداً يكون أفضل.

✅ استخدم الوكلاء المتعددين لـ:
- سير عمل معقد ومتعدد الخطوات
- مهام تحتاج تخصصاً
- التحقق من الجودة (وكيل ناقد)
- سيناريوهات استكشافية/مناظرة
- تنسيق يشبه فريق العمل
❌ تجنب الوكلاء المتعددين لـ:
- مهام بخطوة واحدة
- تصنيف/تلخيص بسيط
- تطبيقات حساسة لزمن الاستجابة
- نشر حساس للتكلفة
- سير عمل خطي محدد
💎

ابدأ ببساطة، وسّع تدريجياً

ابنِ وكيلاً واحداً أولاً. إذا لم يستطع التعامل مع المهمة، أضف متخصصين. أضف وكيلاً ناقداً إذا كانت الجودة مشكلة. هذا النهج التدريجي يمنع الإفراط في الهندسة.

✓

اختبر فهمك

اختبار سريع — 4 أسئلة

1. في نمط المشرف، ماذا يفعل المشرف؟

2. نمط المناظرة هو الأفضل لأي نوع من المشاكل؟

3. ما أهم المخاطر في أنظمة الوكلاء المتعددين؟

4. متى يكون الوكيل الواحد أفضل من الوكلاء المتعددين؟

ملخص الموضوع 16

أنظمة الوكلاء المتعددين تطبّق التخصص على المشاكل المعقدة. أربعة أنماط تسيطر: المشرف (منسّق واحد، عدة متخصصين)، نظير إلى نظير (تعاون مسطّح)، خط الأنابيب (تحويل تسلسلي)، والمناظرة (تحسين جودة تنافسي). بروتوكولات التواصل تحدد كيف ينسّق الوكلاء: تمرير الرسائل، الحالة المشتركة، السبورة، أو قاعدة البيانات. التحديات تشمل عبء التنسيق، الحلقات اللانهائية، المخرجات المتناقضة، وانفجار التكاليف. ليست كل مشكلة تحتاج وكلاء متعددين — الوكلاء المنفردون أبسط وأرخص للمهام المباشرة. استخدم الوكلاء المتعددين للمشاكل المعقدة فعلاً، متعددة الخطوات، حيث يضيف التخصص قيمة.
