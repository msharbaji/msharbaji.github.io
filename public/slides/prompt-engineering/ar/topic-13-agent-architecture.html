هندسة الوكلاء

غوص عميق في الأسس التقنية: التخطيط، والذاكرة، واستخدام الأدوات، وحلقة الاستدلال التي تجعل الوكلاء يعملون على نطاق واسع. افهم كيفية بناء أنظمة وكلاء متينة وجاهزة للإنتاج.

1

الركائز الأربع لهندسة الوكلاء

كل وكيل فعّال يرتكز على أربعة مكونات أساسية. هذه ليست اختيارية — بدون أي واحدة منها، سيفشل وكيلك أو سيتصرف بشكل غير متوقع. لنبنِ نموذجاً ذهنياً لكيفية تفاعلها معاً.

🎯

1. التخطيط والاستدلال

قدرة الوكيل على تقسيم هدف معقد إلى مهام فرعية وتحديد الإجراء التالي. هنا تتألق قدرة النموذج اللغوي على الاستدلال. إنها مرحلة "التفكير".

🧠

الذاكرة

قصيرة المدى (المحادثة)، طويلة المدى (قاعدة بيانات متجهية)، عاملة (مسودة). يحتاج الوكلاء إلى الذاكرة لتجنب الحلقات اللانهائية والحفاظ على السياق.

⚙️

الأدوات

وظائف خارجية يمكن للوكيل استدعاؤها: واجهات برمجة التطبيقات، وقواعد البيانات، والآلات الحاسبة، والبحث على الويب. الأدوات تربط الوكيل بالواقع.

🔄

تنفيذ الإجراءات

الإطار الذي يستدعي الأدوات، ويحلل النتائج، ويعيدها إلى الوكيل. إنها مرحلة "التنفيذ".

💡

مخطط الهندسة

مدخلات المستخدم ← التخطيط (النموذج يقرر ماذا يفعل) ← اختيار الأداة ← تنفيذ الإجراء ← معالجة النتائج ← تحديث الذاكرة ← العودة إلى التخطيط (إذا لم يتحقق الهدف). هذه الدورة هي الوكيل.

2

التخطيط والاستدلال — نمط ReAct

ReAct هو اختصار لـ "Reasoning + Acting" (الاستدلال + التنفيذ). إنه نمط مُثبت حيث يُولّد الوكيل خطوة استدلال (ماذا يفعل)، ثم خطوة إجراء (أي أداة يستخدم)، ثم يراقب النتيجة. هذه الدورة فعّالة بشكل ملحوظ.

Thought
Agent's internal reasoning: "I need to search for current AI agent frameworks to answer this question accurately."
Action
Tool to invoke: web_search("AI agent frameworks 2025")
Observation
Tool result: [search results with links to LangChain, Claude SDK, CrewAI docs]
Python — ReAct Pattern Implementation 
def react_loop(user_input: str) -> str:
    # Build the context with tools available
    available_tools = ["web_search", "calculator", "code_executor"]

    # Keep track of the thought-action-observation history
    history = []

    while not goal_reached:
        # 1. THOUGHT: What should I do?
        thought = llm.generate(
            f"Given {user_input} and my tools {available_tools}, what's my next step?"
        )
        history.append({"type": "thought", "content": thought})

        # 2. ACTION: Execute the tool
        tool_name, tool_args = parse_action(thought)
        result = execute_tool(tool_name, tool_args)
        history.append({"type": "action", "tool": tool_name})

        # 3. OBSERVATION: What happened?
        history.append({"type": "observation", "content": result})

        # Check if we're done
        goal_reached = check_goal(result)

    return synthesize_final_answer(history)

لماذا ينجح نمط ReAct

بفصل التفكير عن التنفيذ، يصبح النموذج أكثر وضوحاً في استدلاله. هذا يجعل تصحيح الأخطاء أسهل ويساعد النموذج على تجنب الأخطاء. إنه أبطأ من الاستجابات المباشرة لكنه أكثر موثوقية بكثير للمهام المعقدة.

3

أنظمة الذاكرة — ثلاث طبقات

يحتاج الوكلاء إلى أنواع مختلفة من الذاكرة ليعملوا بفعالية. فكّر فيها كذاكرة الإنسان: قصيرة المدى (ما تفكر فيه الآن)، طويلة المدى (حقائق تعلمتها منذ زمن)، وعاملة (دفتر ملاحظاتك).

💭

الذاكرة قصيرة المدى

تاريخ المحادثة. الرسائل الأخيرة ونتائج الأدوات. محدودة بنافذة السياق (4 آلاف - 200 ألف توكن). تُمسح في كل جلسة.

📚

الذاكرة طويلة المدى

معرفة دائمة. قواعد بيانات متجهية للبحث الدلالي. حقائق مُتعلَّمة عبر الجلسات. يمكن أن تكون ضخمة (ملايين المستندات).

📝

الذاكرة العاملة

مسودة للمهمة الحالية. نتائج وسيطة، سلاسل الاستدلال، متغيرات الحالة. تُحفظ في الذاكرة قصيرة المدى لكن منفصلة عن المحادثة.

⚙️

أمثلة

قصيرة: آخر 10 رسائل. طويلة: تاريخ العميل في قاعدة بيانات متجهية. عاملة: قائمة المهام الفرعية الحالية، عدد التكرارات.

Python — Memory Architecture Pattern 
class AgentMemory:
    def __init__(self):
        # Short-term: conversation history
        self.conversation = []

        # Long-term: vector database
        self.vector_db = VectorDatabase()

        # Working: current task state
        self.scratchpad = {
            "current_task": None,
            "subtasks_remaining": [],
            "iteration_count": 0,
            "max_iterations": 10
        }

    def add_observation(self, text: str):
        self.conversation.append({"role": "observation", "content": text})
        self.vector_db.add(text)  # Also store for long-term retrieval

    def get_context(self, query: str) -> str:
        # Combine short-term + relevant long-term
        recent = self.conversation[-10:]  # Last 10 messages
        relevant = self.vector_db.search(query, top_k=5)
        return format_context(recent + relevant)

دمج الأدوات — اختيار الأدوات الديناميكي

يحتاج الوكيل إلى معرفة الأدوات المتاحة ومتى يستخدم كل واحدة. يُحل هذا بنمط سجل الأدوات حيث تُوصف كل أداة بمخطط JSON.

Python — Tool Registry Pattern 
TOOL_REGISTRY = {
    "web_search": {
        "description": "Search the web for current information",
        "parameters": {
            "query": {"type": "string", "description": "Search terms"}
        },
        "function": web_search_impl
    },
    "execute_code": {
        "description": "Run Python code safely",
        "parameters": {
            "code": {"type": "string", "description": "Python code to execute"}
        },
        "function": execute_code_impl
    }
}

def execute_tool(tool_name: str, args: dict) -> str:
    if tool_name not in TOOL_REGISTRY:
        raise ValueError(f"Tool {tool_name} not found")

    func = TOOL_REGISTRY[tool_name]["function"]
    try:
        result = func(**args)
        return result
    except Exception as e:
        return f"Error: {str(e)}"

تعريفات الأدوات التصريحية

بتعريف الأدوات في JSON، يستطيع الوكيل قراءة الأوصاف وتقرير أي أداة يستخدم. هكذا يعمل استخدام الأدوات في Claude — يرى النموذج المخطط ويختار بذكاء.

استمرارية الحالة

يجب على الوكلاء طويلي التشغيل حفظ الحالة على القرص أو في قاعدة البيانات. إذا تعطل الوكيل، يمكنه الاستئناف من حيث توقف. هذا أمر حاسم لأنظمة الإنتاج التي تتعامل مع عمليات مكلفة.

6

التعافي من الأخطاء — عندما تسوء الأمور

وكلاء الإنتاج سيفشلون. الأدوات تفشل، والشبكات غير موثوقة، والنماذج ترتكب أخطاء. الوكلاء الممتازون لديهم استراتيجيات للتعافي بأناقة.

استراتيجية إعادة المحاولة ✅
Tool fails → Wait (exponential backoff)
→ Retry up to 3 times → Fallback
to different tool or human
استراتيجية إعادة التخطيط ✅
Tool returns unexpected result
→ Agent re-reasons → Chooses
different tool or approach
التصعيد ✅
Tool fails 3+ times → Mark
as failed → Skip step → Try
next step or ask human
قاطع الدائرة ✅
Tool fails repeatedly → Disable
it temporarily → Use fallback
tools → Monitor for recovery
Python — Error Recovery Pattern 
def execute_with_recovery(tool_name: str, args, max_retries=3):
    retry_count = 0
    wait_time = 1  # seconds, exponential backoff

    while retry_count < max_retries:
        try:
            result = execute_tool(tool_name, args)
            return {"success": True, "result": result}

        except ToolError as e:
            retry_count += 1
            if retry_count < max_retries:
                time.sleep(wait_time)
                wait_time *= 2  # exponential backoff
                continue
            else:
                return {
                    "success": False,
                    "error": str(e),
                    "fallback_tool": "manual_review"
                }
🔑

التدهور التدريجي مقابل الفشل الكامل

الوكلاء الجيدون يتدهورون بأناقة. إذا فشلت أداة البحث، جرّب نسخة مخبأة. إذا فشل الرؤية، اسأل المستخدم. إذا انتهت مهلة تنفيذ الكود، بسّط المشكلة. التدهور التدريجي يبقي الوكلاء مفيدين حتى عندما تتعطل الأمور.

تجميع الكل — الهندسة الكاملة

لنرَ كيف تتصل الركائز الأربع في مثال كامل وعملي. هذا هو النموذج الذهني الذي تحتاجه لبناء وكلاء الإنتاج.

Python — Full Agent Architecture 
class ReasoningAgent:
    def __init__(self):
        self.memory = AgentMemory()
        self.tools = TOOL_REGISTRY
        self.state = {"status": "idle"}
        self.iteration_count = 0

    def run(self, user_goal: str):
        self.state["goal"] = user_goal
        self.state["status"] = "in_progress"

        while (self.iteration_count < 10
               and self.state["status"] == "in_progress"):

            # 1. PERCEIVE: Gather context
            context = self.memory.get_context(user_goal)

            # 2. REASON: What to do next?
            thought = self.llm.generate(
                f"Goal: {user_goal}\nContext: {context}\nWhat's next?"
            )

            # 3. ACT: Execute tool
            tool_name, tool_args = parse_tool(thought)
            result = execute_with_recovery(tool_name, tool_args)

            # 4. LEARN: Update memory and state
            self.memory.add_observation(result)
            self.iteration_count += 1

            # Check if goal is reached
            done = self.llm.judge(
                f"Is this goal achieved? {user_goal}\nEvidence: {result}"
            )
            if done:
                self.state["status"] = "completed"

        return synthesize_answer(self.memory)

رؤى أساسية

هذه الهندسة البسيطة تتضمن كل شيء: التخطيط (الفكر)، واستخدام الأدوات (الإجراء)، والتعافي من الأخطاء، وتتبع الحالة، والتحقق من الهدف. أنظمة الإنتاج الحقيقية تضيف طبقات أكثر (التسجيل، والمراقبة، والتخزين المؤقت)، لكن حلقة العمل الأساسية هذه عالمية.

ملخص الموضوع 13

هندسة الوكلاء ترتكز على أربع ركائز: التخطيط والاستدلال (نمط ReAct)، والذاكرة (قصيرة المدى/طويلة المدى/عاملة)، والأدوات (سجل للوظائف المتاحة)، وتنفيذ الإجراءات (الإطار الذي يستدعي الأدوات بأمان). ReAct (الاستدلال + التنفيذ) يفصل مرحلة التفكير عن مرحلة التنفيذ، مما يجعل الوكلاء أكثر موثوقية. أنظمة الذاكرة تمنع الحلقات اللانهائية وتمكّن إعادة استخدام المعرفة. سجلات الأدوات تتيح للوكلاء اختيار ما يفعلونه بذكاء. إدارة الحالة تمكّن الاستئناف وتصحيح الأخطاء. استراتيجيات التعافي من الأخطاء (إعادة المحاولة، إعادة التخطيط، التصعيد، قاطع الدائرة) تبقي الوكلاء يعملون تحت الضغط.
